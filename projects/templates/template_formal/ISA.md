## Problem

The `template` monorepo has 20 public research-project exemplars (`projects/templates/*`) covering numerical analysis, prose, madlibs, textbooks, autoresearch loops, Active Inference, and more — but none of them demonstrate strong static typing as the load-bearing research subject. Nothing in the roster shows illegal-state-unrepresentable design (ADTs, session-type-shaped protocol state machines, affine-discipline resource handles) applied to a genuinely decentralized domain. Daniel has a real research grounding for exactly this gap: a completed deep-research review (`~/Downloads/deep_research_export/deep_review.md`, 13,739 words, 8 sections, 880 sources) on strongly typed programming, category theory, formal proof assistants, and cognitive modeling — currently sitting outside the repo, unused as a template seed. Separately, there is no exemplar that models a decentralized multiagent system (independent local DB + independent local networking per agent, no shared global state) even though this is directly relevant to Daniel's ant-collective-behavior and Active Inference research.

## Vision

A visitor forks `template_formal`, runs `uv run pytest`, and watches mypy --strict and a real test suite jointly prove that an ant-robot colony simulation cannot enter illegal states — a dropped network message can't corrupt an agent's DB, a consumed transaction handle can't be reused, a handshake can't skip a phase — and then opens the rendered manuscript PDF and finds an honest, citable argument for *why* (grounded in Wadler/Pierce/Honda/Girard/Spivak/Friston, not marketing copy) that reads as scientifically serious, not a typing-features demo reel. The euphoric surprise: the manuscript explicitly draws the line between "mypy proved this" and "this is a runtime discipline, not a proof" — and the tests actually demonstrate both sides of that line with a negative control, rather than merely asserting it.

## Out of Scope

- **No real distributed deployment.** No real TCP/UDP sockets, no real multi-process or multi-host execution, no message broker. The "networking" layer is an in-process, fault-injectable typed message bus; the "database" layer is a real (non-mocked) local SQLite file per agent. This is a decentralization-*shaped* simulation, not a production distributed system.
- **No dependent-type or linear-type soundness claim.** Python has neither. Affine-style handles are a *runtime-guarded discipline*, not a compiler-enforced guarantee. The manuscript states this explicitly; no code comment, docstring, or prose sentence may claim compiler-level linearity.
- **No mandatory formal side-spec.** A Lean 4 or TLA+ specification of the protocol is optional and, if shipped, must be wired to one real runnable check command — never decorative.
- **No UI, no dashboard, no web frontend.** Manuscript + CLI scripts + figures only, matching sibling exemplar shape.
- **No production consensus/Raft/Paxos implementation.** Coordination-layer content is documentary/typed-interface only (a typed "shared pheromone field" substrate), not a full consensus protocol implementation.
- **No mocking framework anywhere in `tests/`** — real SQLite files, real subprocess `mypy --strict` invocations, real in-process objects.

## Principles

- Static typing's honest value in Python is edit-time/CI-time error prevention on structurally representable invariants — never claim runtime enforcement Python's type system does not provide.
- Illegal states should be unrepresentable wherever the type system genuinely allows it; where it doesn't (affine reuse, malformed wire bytes from an untyped boundary), the code must runtime-guard and the manuscript must say so.
- A claim in prose that cannot be pointed at a negative-control test is not a claim the manuscript is entitled to make.
- The ant-robot domain is chosen because bounded local computation, local sensing, and stigmergic (indirect, environment-mediated) coordination are actually true of ant colonies and actually motivate per-agent local DB + local networking — the domain must earn its place scientifically, not decoratively.
- Thin orchestrators, business logic in `src/`, no mocks, real coverage — same discipline as every sibling exemplar.

**Round 14 expansion (cross-vendor audit finds and fixes a real 24-orders-of-magnitude Fisher-exact-test bug, breaking two consecutive clean audit rounds) — added 2026-07-09**
- [x] ISC-114 (Anti, regression-prevention): `colony/stats.py::fisher_exact_test_two_sided`'s two-sided tail must not silently degenerate to the wrong tolerance regime on a highly-separated 2x2 table. A Forge cross-vendor audit of Round 8 (ISC-112's plateau-vs-floor comparison, the perfectly separated `60/60`-vs-`0/60` table) independently recomputed the function's own output three ways (the project function itself, a hand-coded hypergeometric sum, and `scipy.stats.fisher_exact`) and found the project function returned `4.308978969823708e-11` when the true two-sided p-value is `2.070073888186964e-35` — wrong by 24 orders of magnitude. Root cause: the two-sided tail's tolerance was additive (`observed_p + 1e-10`), which is a safe float-equality guard only when `observed_p` is itself around `1e-10` or larger; for this table, `observed_p ≈ 1.035e-35`, so the additive `1e-10` term completely swallowed it, degenerating the inclusion test to "every table with probability at most `1e-10`" (summing a large, wrong swath of the tail) instead of "every table at least as extreme as the observed one." None of the five previously-pinned Fisher-exact test cases (decay-dip `p=0.1187`, ISC-111 extreme-config `p=0.002634`, ISC-113's two seed-base comparisons `p=0.19699`/`p=0.00168`) exercised this failure mode, because all five had `observed_p` comfortably above `1e-10` — the bug was latent and undetected across two full audit rounds (6 and 7, both clean PASS verdicts) plus this round's own fresh-eyes ideation, verify-select, and primary-session independent re-verification, none of which happened to cross-check against an independent oracle (`scipy`) for exactly this reason. Fixed by switching to a relative tolerance (`observed_p * (1.0 + 1e-9)`), independently verified to reproduce all five previously-correct pinned values bit-identically while correcting the sixth (the perfectly-separated table) to the true `2.070073888186964e-35`. The scientific conclusion at the affected call site is unchanged (both the wrong and the correct value are overwhelmingly significant), but a function this manuscript explicitly advertises as "a real hypergeometric computation... not an approximation" owes exactness at every magnitude it is asked to compute, not only the ones its existing test suite happened to probe. Proven by a new dedicated regression test, `tests/colony/test_colony_stats_unit.py::test_fisher_exact_perfectly_separated_table_matches_hand_computed_value`, which pins the corrected value against an independent hand computation (`2/C(120,60)`, not the function's own internal route) specifically for a highly-separated table, so a regression back to an additive tolerance fails this dedicated unit test immediately rather than only surfacing as a silently wrong number several call-sites away in a manuscript quote — precisely how this bug survived undetected for two rounds.
